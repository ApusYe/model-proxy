## S1. Concrete Candidate Error Assertions

### [Assertion 1] Task 3 step 4: `ConfigStore()` init called from `@State` property wrapper may fail Swift 6 strict concurrency because `ConfigStore` is `@MainActor`-isolated but `@State` initializers are not guaranteed to run on @MainActor in all Swift 6 scenarios.

**Verification:** Read Task 8 code (plan line 429): `@MainActor @Observable final class ConfigStore`. Read Task 3 code (plan line 119): `@State private var configStore = ConfigStore()`.

In Swift 6, `@State` property wrappers in an `App` struct use the default initializer expression. The `App` struct `ModelProxyApp` is not explicitly `@MainActor`-annotated, but `App` protocol body is evaluated on the main actor. The `@State` wrapper's `wrappedValue` parameter is evaluated lazily on first access (which happens in the `body` computed property, which is on `@MainActor`). The `ConfigStore.init()` calls `ConfigStore.loadOrCreateDefault()` which is a `static func` on a `@MainActor` class. In Swift 6, calling a `@MainActor`-isolated `init()` from a non-isolated context is a compile error.

However, `App` conformance implies `@MainActor` isolation for `body`, and `@State` default values are typically evaluated in that context. The Swift compiler may or may not flag this depending on the exact version. This is a **medium-risk** concern.

**Result:** Warning -- the assertion is plausible but not definitively provable without compiling. The plan's own note in Task 7 (line 404) acknowledges that `@MainActor` may be needed on `AppConfig`. The same concern applies more directly to `ConfigStore.init()` being called from a `@State` initializer. Plan should note: if a concurrency error arises at `@State private var configStore = ConfigStore()`, annotate `ModelProxyApp` with `@MainActor` explicitly.

### [Assertion 2] Task 8: `ConfigStore.loadOrCreateDefault()` is a `static func` (not `nonisolated`) on a `@MainActor` class, but it is called from `init()`. In Swift 6, `static func` on a `@MainActor` class is `@MainActor`-isolated. Calling it from `init()` is valid because `init()` is also `@MainActor`-isolated. No issue here.

**Verification:** Swift 6 rules: all members of a `@MainActor` class (including `static` members and `init`) inherit `@MainActor` isolation unless marked `nonisolated`.

**Result:** PASS -- `init()` is `@MainActor`, `loadOrCreateDefault()` is `@MainActor` (inherited), call is valid.

### [Assertion 3] Task 3 step 4: `.environment(configStore.config)` passes `AppConfig` (an `@Observable` class) into the environment. SwiftUI environment injection for `@Observable` classes uses `@Environment(AppConfig.self)` on the consumer side. But `StatusPopover` in Task 3 step 5 does NOT declare any `@Environment` property for `AppConfig` -- it is a static placeholder that doesn't read config. This means the environment injection is unused in Phase 1, but will not cause a compile error.

**Verification:** Read Task 3 StatusPopover code (plan lines 146-161): no `@Environment` property declared. Read Task 3 ModelProxyApp code (plan line 125): `.environment(configStore.config)`.

**Result:** PASS -- no compile error. The environment injection is set up for future phases. However, there is a subtle issue: `.environment(configStore.config)` injects an `AppConfig` instance. Future views will need `@Environment(AppConfig.self) var config` to consume it. This is correct SwiftUI `@Observable` environment usage.

### [Assertion 4] Task 8: `JSONEncoder.pretty` is defined as a `static let` on a `private extension JSONEncoder`. In Swift 6 strict concurrency, `JSONEncoder` is not `Sendable`. A `static let` of a non-Sendable type triggers a warning/error about shared mutable state.

**Verification:** `JSONEncoder` is a class (reference type) and is not marked `Sendable` in Foundation. A `static let` property creates a global shared instance. In Swift 6 with strict concurrency, this would produce: `Static property 'pretty' is not concurrency-safe because non-'Sendable' type 'JSONEncoder' may have shared mutable state`.

However, `ConfigStore` is `@MainActor`-isolated. The `static let pretty` is in a `private extension JSONEncoder`, not on `ConfigStore`. It's a module-level static, accessible from any isolation domain. In Swift 6, this is a concurrency warning/error.

**Result:** FAIL -- `static let pretty: JSONEncoder` will likely produce a Swift 6 strict concurrency error. The plan should either: (a) make it `nonisolated(unsafe) static let pretty` (Swift 6 escape hatch for known-safe statics), or (b) create a new encoder instance each time in `save()` and `loadOrCreateDefault()`, or (c) mark it with `@MainActor` since it's only used from `@MainActor` context.

## Additional Findings

### Finding 1: TokenStats.add() double dictionary lookup

In Task 6 (plan lines 311-312), `TokenStats.add()` performs two separate subscript operations on `records[vendorID, default: [:]]`:
```swift
records[vendorID, default: [:]][modelID, default: ModelTokenRecord()].inputTokens += input
records[vendorID, default: [:]][modelID, default: ModelTokenRecord()].outputTokens += output
```

The first line may create the default nested dictionary and record, then the second line re-accesses `records[vendorID, default: [:]]` which creates a NEW default empty dictionary if the first mutation did not persist correctly. However, with Swift's value-type mutation through subscript, the first line does mutate `records` in place. The second line then accesses the already-mutated dictionary. This is correct but wasteful -- two dictionary lookups instead of one. Not a bug, but could be cleaner with a single mutation block. This is a style concern, not a must-revise.

**Result:** PASS (functional correctness OK; minor style issue)

Future phases that need to read config from the NIO proxy (which runs on NIO event loops, not main actor) will need a `Sendable` snapshot or explicit `@MainActor` hop. Not blocking for Phase 1.

### [DP-001] App Sandbox configuration (blocking)

**Context:** The project has `ENABLE_APP_SANDBOX = YES` (Xcode default). This conflicts with the project constraint that config lives at `~/Library/Application Support/ModelProxy/config.json` (not the sandboxed container path), and Phase 2 will need localhost server binding. **Options:**
- A: Disable App Sandbox entirely (`ENABLE_APP_SANDBOX = NO`) -- simplest for a personal-use tool; allows direct filesystem access and localhost binding without entitlements
- B: Keep sandbox, add entitlements (network.server, network.client, file read/write) and update config path documentation to reflect sandboxed location -- more secure but adds complexity for a single-user tool
**Recommendation:** A -- this is a personal-use localhost proxy; sandbox adds configuration overhead with no security benefit for the use case.
