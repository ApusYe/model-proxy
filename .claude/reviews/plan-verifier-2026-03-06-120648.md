**Verdict:** must-revise. Task 3 step 3 needs correction:

2. Add `com.apple.security.network.server` (required for proxy listener).
3. Add `com.apple.security.network.client` (required for upstream API calls).

### [Assertion 2] Task 2: Plan says to set `SWIFT_STRICT_CONCURRENCY = complete` but the project already has `SWIFT_DEFAULT_ACTOR_ISOLATION = MainActor` and `SWIFT_APPROACHABLE_CONCURRENCY = YES` (Xcode 16+ Swift 6 defaults). Plan does not mention these existing settings, which significantly affect concurrency behavior.

**Verification:** Read `project.pbxproj` lines 416-417 (Debug) and 448-449 (Release):
```
SWIFT_APPROACHABLE_CONCURRENCY = YES;
SWIFT_DEFAULT_ACTOR_ISOLATION = MainActor;
```

These are Xcode 16 Swift 6 settings. `SWIFT_DEFAULT_ACTOR_ISOLATION = MainActor` means all types default to `@MainActor` isolation unless explicitly opted out. `SWIFT_APPROACHABLE_CONCURRENCY = YES` relaxes some Sendable requirements.

**Result:** This is not strictly an error but a significant omission. With `SWIFT_DEFAULT_ACTOR_ISOLATION = MainActor`:
- All structs and classes in the plan (Vendor, ClientConfig, TokenStats, etc.) will be implicitly `@MainActor`. For pure data models marked `Sendable`, this may cause warnings or errors because `@MainActor` types are not automatically `Sendable`.
- The explicit `@MainActor` on `ConfigStore` is redundant (but harmless).
- The `nonisolated(unsafe)` on `JSONEncoder.pretty` is correct regardless.
- `TokenStats` is marked `Sendable` but with default MainActor isolation, this creates a potential conflict. The `mutating func add()` would be MainActor-isolated, which is fine for now but contradicts the stated intent of it being managed by "a future TokenStatsStore actor".

**Verdict:** advisory. The plan should acknowledge the existing `SWIFT_DEFAULT_ACTOR_ISOLATION = MainActor` setting in Task 2 and note that:
- Pure data model structs (`Vendor`, `ClientConfig`, `ModelTokenRecord`, `DailyTokenSnapshot`) should work fine since they are `Sendable` value types.
- `TokenStats` may need `nonisolated` annotation in future phases when moved to a non-MainActor context.
- The plan should state whether to keep or change `SWIFT_DEFAULT_ACTOR_ISOLATION` and `SWIFT_APPROACHABLE_CONCURRENCY`.

## Decisions

### [DP-001] Sandbox mode vs. non-sandbox for config file path (blocking)

**Context:** The project has `ENABLE_APP_SANDBOX = YES` (Xcode default). This redirects `~/Library/Application Support/` to a sandboxed container path. The CLAUDE.md constraint specifies "API keys stored in ~/Library/Application Support/ModelProxy/config.json", which implies the user-visible path. Additionally, this is a personal-use developer tool where sandbox restrictions on localhost binding require explicit entitlements.

**Options:**
- A: Keep sandbox enabled, add network entitlements, update all path references to sandboxed path -- follows Apple defaults, but config.json is buried in Containers directory and harder to manually edit
- B: Disable sandbox (`ENABLE_APP_SANDBOX = NO`), keep hardened runtime -- config.json at user-visible `~/Library/Application Support/ModelProxy/config.json` as specified in CLAUDE.md, no entitlements needed for network or file access, simpler for a personal developer tool

**Recommendation:** B -- this is a personal-use localhost-only tool where sandbox adds friction (hidden config path, entitlement requirements for localhost binding) without meaningful security benefit. The CLAUDE.md constraint explicitly names the non-sandboxed path.
