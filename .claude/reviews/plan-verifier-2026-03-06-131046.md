## S1. Concrete Candidate Error Verification

### [Assertion 1] Task 6: `AppConfig` is not `Sendable` but is captured across concurrency boundaries in `StatusPopover.task { }`, causing a Swift 6 strict concurrency error.

**Verification:** Read `/Users/norvyn/Code/Projects/ModelProxy/ModelProxy/Models/AppConfig.swift:8` -- `final class AppConfig: Codable {` -- no `Sendable` conformance. The plan's `StatusPopover` (Task 6, step 2) reads `config` from `@Environment` on MainActor, then passes it to `proxyServer.start(port:config:)` which runs in an async context. `ProxyServer.start` accepts `AppConfig` (line 445 of plan). Since `AppConfig` is a non-Sendable reference type (`@Observable final class`), passing it across actor boundaries violates Swift 6 strict concurrency.

Furthermore, `RoutingSnapshot.init(from: AppConfig)` (Task 1) reads `config.vendors` and `config.clients` -- both mutable `var` properties on a non-Sendable class. In Swift 6 strict concurrency mode, the compiler will reject this crossing.

**Result:** The assertion is valid. `AppConfig` is `@Observable final class` (not Sendable). Passing it to `ProxyServer.start` from a `@MainActor` context and then reading its properties in `RoutingSnapshot.init` crosses isolation boundaries.

**Plan needs:** Either (a) make `ProxyServer.start` accept a `RoutingSnapshot` directly (built on MainActor before crossing), or (b) make `AppConfig` conform to `Sendable` / `@unchecked Sendable`, or (c) mark `ProxyServer` as `@MainActor` (but the plan explicitly says it is NOT `@MainActor`). Option (a) is cleanest -- build the snapshot on MainActor, pass the `Sendable` struct across.

### [Assertion 2] Task 4: `ProxyServer` is `@Observable final class` with `@unchecked Sendable`, but its mutable properties (`isRunning`, `lastError`, `boundPort`, internal NIO refs) are mutated from `start()`/`stop()` which can be called from any isolation domain, while SwiftUI reads them on MainActor -- no synchronization mechanism exists.

**Verification:** Plan line 426: `final class ProxyServer: @unchecked Sendable`. Properties `isRunning`, `lastError`, `boundPort` are `var` (lines 430-432). `start()` and `stop()` are `async` methods with no actor isolation. SwiftUI observes these via `@Observable` on MainActor. Without actor isolation or a lock, concurrent reads (SwiftUI on MainActor) and writes (`start`/`stop` on cooperative pool) create a data race.

**Result:** The assertion is valid. `@unchecked Sendable` on a class with unprotected mutable state is a data race. The plan explicitly chose not to use an actor for `ProxyServer` (to use `@Observable`), but provides no alternative synchronization.

**Plan needs:** Either (a) dispatch property mutations back to MainActor using `@MainActor` on the class or on `start()`/`stop()`, or (b) use `DispatchQueue`/lock for property access, or (c) make `ProxyServer` an actor (but `@Observable` on actors has limitations). The simplest fix: annotate `ProxyServer` as `@MainActor` -- since `start()` is `async`, the NIO work happens off-main via `await`, and only property mutations happen on MainActor. This is the standard SwiftUI pattern.

### [Assertion 4] Task 3/5: `channel.writeAndFlush(NIOAny(...))` is called from an async `Task` (off the event loop). NIO channels require writes to happen on the event loop thread, and `writeAndFlush` on `Channel` (not `ChannelHandlerContext`) is thread-safe -- but the `.get()` pattern used (`try await channel.writeAndFlush(...).get()`) may not exist on the async Channel API.

**Verification:** In SwiftNIO 2.x, `Channel.writeAndFlush(_:)` returns `EventLoopFuture<Void>`. The plan calls `.get()` which is the `EventLoopFuture` bridge to async (available since NIO 2.40+). However, there is a subtlety: `Channel.writeAndFlush` accepts `NIOAny`, but the `Channel` protocol's `writeAndFlush` method signature is `func writeAndFlush(_ data: NIOAny) -> EventLoopFuture<Void>`. The plan wraps `HTTPServerResponsePart` in `NIOAny()` explicitly. This should work because the pipeline has `HTTPServerResponseEncoder` (added by `configureHTTPServerPipeline`), which expects `HTTPServerResponsePart` as outbound.

**Result:** Assertion does not hold. The pattern is correct for NIO 2.95.0. `Channel.writeAndFlush` is thread-safe, and `.get()` bridges to async correctly.

### [Runtime Regression 1] Proxy starts only when popover is opened

**Assumed regression:** The proxy server starts in `StatusPopover.task { }`. The `.task` modifier runs when the view appears. For `MenuBarExtra` with `.menuBarExtraStyle(.window)`, the popover content is lazily created -- `StatusPopover` does not appear until the user clicks the menu bar icon. This means the proxy will NOT start on app launch; it starts only after the first click on the menu bar icon.

**Operation path:** App launches -> menu bar icon appears -> user has NOT clicked it -> `StatusPopover` not created -> `.task` not executed -> proxy not running -> curl to localhost:8080 returns "connection refused"

**Plan coverage:** Not covered. The plan's Task 6 step 1 even contains a comment (lines 632-639) acknowledging that SwiftUI menu bar apps lack `applicationDidFinishLaunching`, and says "the StatusPopover triggers startup on first appearance." But this means the proxy is NOT running until the user clicks the icon, which is a significant UX regression -- a transparent proxy that requires manual activation is not transparent.

**Plan needs:** Start the proxy independently of the popover appearing. Options: (a) use `NSApplicationDelegate` via `@NSApplicationDelegateAdaptor` to start the proxy in `applicationDidFinishLaunching`, (b) add a `.task` modifier at the `MenuBarExtra` level (not inside the popover content), or (c) use an `onAppear` at a level that triggers on app launch.

## Decisions

### [DP-001] URL composition convention for baseURL + request URI (blocking)

**Context:** The default Anthropic vendor's `baseURL` is `"https://api.anthropic.com/v1"` (from `AppConfig.makeDefault()`). Incoming requests use URI `/v1/messages`. Concatenating them produces a doubled path `/v1/v1/messages`. The plan must define how base URLs and request URIs compose.

**Options:**
- A: `baseURL` excludes version path (e.g., `"https://api.anthropic.com"`) -- forwarder concatenates `baseURL + head.uri` directly. Requires changing default config and documenting the convention for all vendors.
- B: `baseURL` includes version path (current convention) -- forwarder strips the matching prefix from `head.uri` before concatenating. More complex; must handle mismatched prefixes.
- C: `baseURL` is the full endpoint root (e.g., `"https://api.anthropic.com/v1"`) and `head.uri` is path-rewritten to strip `/v1` prefix. Requires knowledge of which path segments belong to the "version prefix."

**Recommendation:** A -- simplest and least error-prone. Change default `baseURL` to `"https://api.anthropic.com"` and document that `baseURL` should be the scheme+host (optionally with a non-versioned path prefix like `/compatible-mode` for DashScope).

### [DP-002] Proxy startup mechanism (blocking)

**Context:** The plan starts the proxy in `StatusPopover.task { }`, which only fires when the user clicks the menu bar icon. A transparent proxy must start on app launch without user interaction.

**Options:**
- A: `@NSApplicationDelegateAdaptor` with `applicationDidFinishLaunching` -- proven pattern for menu bar apps that need lifecycle hooks.
- B: Background `Task` in `ModelProxyApp.init()` -- less conventional but avoids AppDelegate.
- C: Keep current approach, document that user must click icon first -- unacceptable for a "transparent" proxy.

**Recommendation:** A -- standard macOS pattern, cleanly separates proxy lifecycle from UI lifecycle.
