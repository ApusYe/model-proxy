### [Assertion 3] Task 6: `StatusPopover` creates its own `@State private var proxyServer = ProxyServer()` while `ModelProxyApp` also creates `@State private var proxyServer = ProxyServer()`. Two independent `ProxyServer` instances exist; neither is connected to the other. The app-level instance is unused and the popover-level instance is not accessible from app-level lifecycle (e.g., for graceful shutdown on termination).

**Verification:** Plan lines 596-626 show `ModelProxyApp` declaring `@State private var proxyServer = ProxyServer()` but never using it -- the comment at line 625 says "the proxy is started from StatusPopover.swift via the environment." Plan lines 636-667 show `StatusPopover` declaring its own `@State private var proxyServer = ProxyServer()`. The app-level `proxyServer` is dead code.

**Recommendation:** Remove the unused `proxyServer` from `ModelProxyApp`. Add an `NSApplicationDelegateAdaptor` or use `@Environment(\.scenePhase)` to call `proxyServer.stop()` before termination. The plan acknowledges this is "Phase 5" scope, so this is a recommendation, not must-revise.

### [Assertion 4] Task 3: `ProxyChannelHandler` is marked `@unchecked Sendable` but has mutable stored properties (`requestHead`, `bodyBuffer`) accessed from the NIO event loop thread via `channelRead`. Under Swift 6 strict concurrency, the `Task { }` closure captures `self.router` and `self.httpClient` (both fine) but the handler itself is accessed from the event loop thread without synchronization. The `@unchecked Sendable` annotation suppresses the warning, but if NIO ever calls `channelRead` from a different thread context (e.g., during pipeline modification), the mutable state is unprotected.

**Verification:** Plan lines 209-263 show `ProxyChannelHandler` with `@unchecked Sendable`, mutable `requestHead` and `bodyBuffer`. NIO channel handlers are guaranteed to be called from the same event loop thread for a given channel, so concurrent access does not occur for a single connection. The `Task { }` at line 242 captures only local `let` copies of `channel`, `router`, `httpClient` -- not the handler's mutable state.

**Result:** The assertion does NOT hold. NIO's threading model guarantees single-threaded access per channel handler instance. The `@unchecked Sendable` is necessary because the handler is created on one thread and passed to the channel pipeline on another, but after that all access is single-threaded. The `Task` closure captures only local copies. This is correct.

### [Assertion 5] Task 5: `ResponseRelay.relay` strips `transfer-encoding` from upstream response headers (plan line 541) but does not add `Content-Length` for non-streaming responses. Without either `Transfer-Encoding: chunked` or `Content-Length`, HTTP/1.1 clients may not know when the response body ends (for non-streaming responses where connection close is the only signal).

**Verification:** Plan lines 538-543 strip `transfer-encoding` and `connection`. Plan line 570 closes the channel after each response. For HTTP/1.1, closing the connection is a valid way to signal end-of-body. However, this breaks HTTP keep-alive -- every request results in a connection close. For curl this is fine; for a real client like Claude Code that may send many requests in sequence, each request pays the TCP connection setup cost.

**Result:** The assertion is PARTIALLY CONFIRMED. The plan explicitly acknowledges this at lines 567-570: "For simplicity in Phase 2, always close after each response. Phase 3+ can add keep-alive support." So this is by design for Phase 2. However, stripping `transfer-encoding` while the upstream may be sending chunked encoding means the response relay writes raw chunked-encoded bytes (including chunk size headers like `1a\r\n`) into a response that claims to be non-chunked. If the upstream sends `Transfer-Encoding: chunked`, the relay must either (a) preserve the header so the client decodes chunks, or (b) decode chunks before relaying. Since `AsyncHTTPClient` handles chunked decoding internally and provides decoded body bytes, the `upstreamResponse.body` async sequence already yields decoded bytes. So stripping `Transfer-Encoding: chunked` is correct -- the bytes are already decoded.

**Result updated:** Assertion does NOT hold. `AsyncHTTPClient` decodes chunked encoding internally. The relay receives decoded bytes and correctly omits `Transfer-Encoding`. Connection close signals end of body. This is correct for Phase 2.

## S2. Failure Reverse Reasoning

### [Compile Failure 1]

**Hypothetical failure:** `ProxyServer` is marked `@MainActor @Observable final class` (plan line 413-414). It is NOT an actor (despite the plan text at line 400 saying "an `@Observable` actor"). `@MainActor` class + `@Observable` should work. However, `ProxyServer.start()` is `async` and calls `bootstrap.bind(...).get()` which blocks the calling thread. Since `ProxyServer` is `@MainActor`, `start()` runs on the main actor. Calling `.get()` on an `EventLoopFuture` from within an async context should use `await`, and the plan does use `try await ... .get()`. This is fine -- `.get()` returns a future that the async runtime can await without blocking the main thread.

**Plan coverage:** Covered. The `await` keyword ensures the main actor is not blocked.

### [Compile Failure 2]

**Hypothetical failure:** `channel.writeAndFlush(NIOAny(...))` in `ProxyForwarder.sendError` (plan lines 373-381) -- in modern SwiftNIO, `Channel.writeAndFlush` returns `EventLoopFuture<Void>`, and calling `.get()` in an async context with `try await` is correct. However, `HTTPServerResponsePart` must match the channel pipeline's expected outbound type. The pipeline is configured with `configureHTTPServerPipeline` (plan line 453), which adds `HTTPResponseEncoder`. The encoder expects `HTTPServerResponsePart` as inbound. But `channel.writeAndFlush(NIOAny(...))` bypasses the handler-level type system -- it writes directly to the channel. This should work because the encoder is in the pipeline and will encode the `HTTPServerResponsePart`.

**Plan coverage:** Covered. The NIO pipeline handles type encoding.

### [Compile Failure 3]

**Hypothetical failure:** `ProxyServer` at plan line 463 calls `bootstrap.bind(host:port:)` which returns `EventLoopFuture<Channel>`. In SwiftNIO 2.95.0, `ServerBootstrap.bind(host:port:)` returns `EventLoopFuture<Channel>`. Calling `.get()` converts it to async. This should compile.

**Plan coverage:** Covered.

### [Runtime Regression 2]

**Hypothetical regression:** The `.task` modifier on `StatusPopover` (plan line 660) fires every time the popover appears (user clicks menu bar icon). If the user clicks the menu bar icon, dismisses the popover, and clicks again, `.task` fires again, calling `proxyServer.start()`. The `start()` method has a `guard !isRunning` check (plan line 434), so the second call is a no-op. This is safe.

**Plan coverage:** Covered by the guard.

### [Runtime Regression 3]

**Hypothetical regression:** `StatusPopover` uses `.task { }` which is tied to the view's lifecycle. When the popover is dismissed (menu bar icon clicked again to close), the task may be cancelled by SwiftUI's cooperative task cancellation. If `proxyServer.start()` is still awaiting `bootstrap.bind()`, the task cancellation would propagate, and the `try await bootstrap.bind(...).get()` would throw `CancellationError`. This would hit the generic catch block (plan line 475), set `lastError`, and the server would not start.

**Operation path:** User clicks menu bar icon -> popover appears -> `.task` fires -> `start()` begins awaiting bind -> user clicks away (popover dismissed) -> SwiftUI cancels the task -> `CancellationError` -> server fails to start.

**Plan coverage:** NOT covered. The `.task` modifier's cooperative cancellation can race with the server bootstrap. This is a real risk: if the user clicks the menu bar icon and then immediately clicks elsewhere, the server startup may be cancelled.

**Recommendation:** Use `Task { }` (detached or unstructured) instead of `.task` modifier, or move server startup to app init / `NSApplicationDelegateAdaptor.applicationDidFinishLaunching`. This ensures startup is not tied to view lifecycle.

## Decisions

### [DP-001] URL path convention for Vendor.baseURL (blocking)

**Context:** The existing `AppConfig.makeDefault()` sets Anthropic vendor `baseURL` to `"https://api.anthropic.com/v1"` (includes path). The plan's URL construction concatenates `baseURL + head.uri` where `head.uri` includes `/v1/messages`. This produces doubled paths. The plan must decide whether `baseURL` includes the API version path or not.

**Options:**
- A: `baseURL` is the root (e.g., `"https://api.anthropic.com"`) and `head.uri` provides the full path. Requires updating `AppConfig.makeDefault()` and documenting the convention. Simple URL construction.
- B: `baseURL` includes the version path (e.g., `"https://api.anthropic.com/v1"`) and URL construction strips the overlapping prefix from `head.uri`. More complex but preserves existing config format.

**Recommendation:** A -- simpler, less error-prone. The proxy is a transparent forwarder; the client's URI already includes the full path.

### [DP-002] Server startup lifecycle location (recommended)

**Context:** The plan starts the proxy server inside `StatusPopover.task { }`, which is tied to the popover view lifecycle. SwiftUI may cancel this task if the popover is dismissed before startup completes.

**Options:**
- A: Keep `.task` on `StatusPopover` -- works if startup is fast (bind is typically sub-millisecond), accept small race window.
- B: Move startup to `ModelProxyApp` using `NSApplicationDelegateAdaptor.applicationDidFinishLaunching` -- decouples from view lifecycle, guaranteed to run once.
- C: Use `.onAppear` with an unstructured `Task { }` -- not cancelled by SwiftUI, but fires on every appearance (guard handles idempotency).

**Recommendation:** B -- cleanest separation of concerns, no cancellation risk, server lifecycle is app-scoped not view-scoped.
