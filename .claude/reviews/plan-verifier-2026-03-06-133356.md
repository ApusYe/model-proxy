## Decisions

### [DP-001] Proxy startup timing (blocking)

**Context:** The plan starts the proxy inside `StatusPopover.task { }`, which only fires when the user opens the menu bar popover. This means the proxy is unavailable until the user clicks the icon, breaking clients that expect the proxy to be ready on app launch.

**Options:**
- A: `NSApplicationDelegateAdaptor` -- add an app delegate that starts the proxy in `applicationDidFinishLaunching`. Proven approach for lifecycle events in menu bar apps. Adds ~15 lines.
- B: Lift `ProxyServer` to `ModelProxyApp` and use an initializer or `init()` side-effect -- not idiomatic SwiftUI, may cause timing issues with `ConfigStore` initialization.
- C: Use `onChange(of: configStore.config)` or similar to trigger startup from the `Scene` body -- fragile, depends on observation timing.

**Recommendation:** A -- `NSApplicationDelegateAdaptor` is the standard pattern for menu bar apps that need startup logic before any view appears.

### [DP-002] baseURL convention: with or without version path (blocking)

**Context:** The plan changes Anthropic's default baseURL to strip `/v1`, but existing doc comments and test data show baseURLs that include version paths (e.g., `/compatible-mode/v1`). The URL construction (`baseURL + head.uri`) depends on a consistent convention.

**Options:**
- A: baseURL = domain + base path only, never includes version -- all vendors must be configured this way. Incoming request URI (e.g., `/v1/messages`) is appended directly. Simple, but requires updating all existing examples and documentation.
- B: baseURL = full API root including version path -- the proxy strips the leading `/v1` from `head.uri` before appending. More tolerant of existing configs, but adds parsing complexity.

**Recommendation:** A -- simpler, and since this is Phase 2 with no existing user configs, now is the right time to establish the convention cleanly.
