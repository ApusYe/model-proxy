## S1. Concrete Candidate Error Verification

### [Assertion 1] Task 6: `.task {}` is a View modifier, not a Scene modifier -- compile error on MenuBarExtra

**Claim:** The plan's `ModelProxyApp.swift` (Task 6 step 1, lines 624-636) applies `.task { ... }` after `.menuBarExtraStyle(.window)` on the `MenuBarExtra` scene. In SwiftUI, `.task {}` is defined on `View`, not on `Scene`. `MenuBarExtra` conforms to `Scene`, and `.menuBarExtraStyle(.window)` returns `some Scene`. Applying `.task {}` to a `Scene` value will produce a compile error: "Value of type 'some Scene' has no member 'task'".

**Verification:** SwiftUI's `task(priority:_:)` modifier is declared as an extension on `View` (see `SwiftUI.View.task`). There is no corresponding `Scene.task` modifier. The plan's code structure:

```swift
MenuBarExtra("ModelProxy", ...) { ... }       // Scene
    .menuBarExtraStyle(.window)                // some Scene
    .task { ... }                              // ERROR: Scene has no .task
```

**Result:** FAIL -- assertion holds. This is a compile error. The plan must use a different mechanism to start the proxy at app launch. The `.task` modifier can only be applied to a `View` inside the scene content, or the plan must use `NSApplicationDelegateAdaptor` for startup.

### [Runtime Regression 3] Task 5: `ResponseRelay` strips `transfer-encoding` but does not set chunked encoding on downstream

**Hypothetical regression:** When the upstream response uses `transfer-encoding: chunked`, `ResponseRelay` strips this header (plan line 553). The relay then writes individual body chunks via `writeAndFlush`. NIO's HTTP server pipeline may or may not automatically add `transfer-encoding: chunked` to the outgoing response. If NIO does not add it, the client receives a response with neither `content-length` nor `transfer-encoding`, which may cause some HTTP clients to fail to detect end-of-response.

**Verification:** NIO's `HTTPServerPipelineHandler` (added via `configureHTTPServerPipeline` in Task 4 line 465) automatically handles HTTP framing. When the server writes `.head` without `content-length`, followed by multiple `.body` parts, followed by `.end`, NIO adds `transfer-encoding: chunked` automatically. This is standard NIO behavior.

**Result:** Covered by NIO's HTTP pipeline framing.

## Decisions

### [DP-001] Proxy startup and shutdown mechanism (blocking)

**Context:** The plan's `.task {}` on `MenuBarExtra` Scene will not compile. A different approach is needed for startup. This also needs to address graceful shutdown (must-revise #3).

**Options:**
- A: `NSApplicationDelegateAdaptor` -- add an `AppDelegate` with `applicationDidFinishLaunching` (start proxy) and `applicationWillTerminate` (stop proxy). Standard pattern for menu bar apps. Adds ~25 lines.
- B: Place `.task {}` on the `StatusPopover` view inside the `MenuBarExtra` content -- but `.window` style evaluates content lazily, so proxy won't start until user opens the popover.
- C: Use `@Observable` lifecycle observation or `NotificationCenter` -- non-standard, fragile.

**Recommendation:** A -- `NSApplicationDelegateAdaptor` is the proven approach for menu bar apps that need startup/shutdown logic independent of view lifecycle.

### [DP-002] Config migration for existing users (recommended)

**Context:** Phase 1 users have `baseURL: "https://api.anthropic.com/v1"` in their persisted config.json. The new URL construction will produce doubled paths for these users.

**Options:**
- A: Add a migration in `ConfigStore` that strips trailing `/v1` from all vendor baseURLs on load. Simple, handles the known case.
- B: Handle in `ProxyForwarder.forward` by detecting `head.uri` starting with the same prefix as the baseURL tail (e.g., both end/start with `/v1`). More general but adds parsing complexity.
- C: Document the change and require users to manually edit config.json. Simplest but poor UX.

**Recommendation:** A -- one-time migration in `ConfigStore` is clean and handles the known case without runtime overhead.
