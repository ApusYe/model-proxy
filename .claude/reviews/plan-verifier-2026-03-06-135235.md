### [Assertion 5] Task 3: `ProxyChannelHandler` is marked `@unchecked Sendable` and has mutable properties (`requestHead`, `bodyBuffer`). It is accessed from both the NIO event loop (in `channelRead`) and potentially from the `Task {}` closure. Does the `Task` closure capture any mutable handler state?

**Verification:** Read plan Task 3 code, `.end` case:
```swift
case .end:

    let channel = context.channel
    let router = self.router
    let httpClient = self.httpClient
    Task {
        await ProxyForwarder.forward(
            head: head,        // value type copy
            body: body,        // value type copy (ByteBuffer is CoW)

**Result:** Assertion does not hold. No mutable handler state is captured in the closure. The `@unchecked Sendable` is safe here because NIO guarantees single-threaded access per channel.

If a client (e.g., Claude Code) sends a request before the user clicks the icon, the request fails with "connection refused" because the proxy is not yet listening.

### [Runtime Regression 2] Task 6: The plan replaces `StatusPopover` completely, removing the `#Preview` macro at line 17-19 of the current file. The new `StatusPopover` uses `@Environment(AppConfig.self)` and `@Environment(ProxyServer.self)`, which means previews would crash without providing these environment values.

**Plan coverage:** Not mentioned. Minor issue -- previews are broken but this is Phase 2 scaffolding. Not blocking.

### [Runtime Regression 3] Task 3/5: Both `ProxyForwarder.sendError` and `ResponseRelay.relay` call `channel.close()` after completing. If two requests arrive on the same connection (HTTP pipelining or keep-alive before the first response completes), the second request's `channelRead` fires after `.end` resets `requestHead`/`bodyBuffer`, but the channel might be closed by the first request's response relay before the second request completes.

**Plan coverage:** The plan explicitly states: "For simplicity in Phase 2, always close after each response. Phase 3+ can add keep-alive support." This means HTTP pipelining is not supported in Phase 2. Since the proxy always sends `Connection: close` in error responses (line 380) but does NOT send `Connection: close` in normal responses (Task 5), clients may attempt keep-alive. However, the `channel.close()` at line 582 of ResponseRelay ensures the connection is terminated. This is acceptable for Phase 2.

**Potential issue:** The normal response relay (Task 5) does not add a `Connection: close` header to the forwarded upstream response. If the upstream response says `Connection: keep-alive`, the client sees keep-alive but the proxy closes the connection anyway. This could cause client-side errors on the *next* request attempt on the same TCP connection. The fix is to add `Connection: close` to all response headers in ResponseRelay.

**Severity:** Recommended improvement, not must-revise for Phase 2.

## Decisions

### [DP-001] Add `Connection: close` to all proxied responses (recommended)

**Context:** Phase 2 always closes connections after each response, but the response relay does not signal this via headers. Clients may interpret upstream `Connection: keep-alive` and attempt reuse, getting reset errors.
**Options:**
- A: Add `Connection: close` header in ResponseRelay -- 1 line change, prevents client confusion
- B: Leave as-is, rely on TCP close -- works but causes occasional client-side "connection reset" warnings
**Recommendation:** A -- single line, eliminates a class of client-side errors during development.
