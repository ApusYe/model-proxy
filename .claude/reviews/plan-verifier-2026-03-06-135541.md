### [Assertion 5] Task 5: Stripping `Transfer-Encoding` from upstream response but not setting it on the proxied response may cause incorrect framing for non-streaming responses

**Claim:** `ResponseRelay` strips `transfer-encoding` from upstream response headers (line 553). For streaming (SSE) responses, NIO's `HTTPResponseEncoder` will automatically add `Transfer-Encoding: chunked` since there's no `Content-Length`. For non-streaming responses where the upstream sent `Content-Length`, the header is preserved and NIO will send a fixed-length response. However, if the upstream used `Transfer-Encoding: chunked` for a non-streaming response (some servers do this), the plan strips the header but NIO will re-add chunked encoding. The original `Content-Length` would be absent.

**Verification:** The plan's `ResponseRelay` (plan line 550-555) strips `transfer-encoding` and `connection`. NIO's `HTTPResponseEncoder` will determine framing based on whether `Content-Length` is present in the response head. If `Content-Length` is present, it uses fixed-length. Otherwise, it uses chunked. This is correct behavior for a proxy.

**Result:** NOT FAILED -- NIO handles this correctly. The proxy strips hop-by-hop headers and lets NIO re-frame. Standard proxy behavior.
