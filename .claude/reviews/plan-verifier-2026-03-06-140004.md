**Primary data flow:**
```

```

No concurrent mutation risk -- snapshot is immutable after construction.

## Decisions

### [DP-001] Passthrough API key semantics (blocking)

**Context:** The passthrough routing path always uses the caller's API key, ignoring the vendor's configured key. Task 8's test instructions are inconsistent with this behavior. This is a design choice that affects how users configure the proxy.

**Options:**
- A: Keep current behavior (caller's key for passthrough) -- the proxy is truly transparent for unmapped models. Users must send valid keys from the client. The vendor's key in config.json is only used for explicitly mapped models. Revise Task 8 step 2 to send a real key.
- B: Use the vendor's configured key for passthrough when `defaultVendorID` is set and the vendor has a non-empty `apiKey` -- this means the proxy substitutes the key for all requests, even unmapped ones. More user-friendly (client can send any dummy key) but less "transparent proxy."

**Recommendation:** A -- true passthrough aligns with the project's "transparent local API proxy" description. Revise the test instructions, not the routing logic.

### [DP-002] Task 8 step 4 prefix-match test (recommended)

**Context:** Task 8 step 4 tests prefix-match routing via `modelPatterns`, but Phase 2 only implements exact-match via `modelMappings`. The plan acknowledges this (line 777) but the test step remains.

**Options:**
- A: Remove Task 8 step 4 entirely (prefix matching is Phase 3+).
- B: Rewrite step 4 to test exact-match routing via `modelMappings` instead.

**Recommendation:** B -- testing exact-match routing is valuable; rewrite the step to add a `modelMappings` entry and verify it routes correctly.
