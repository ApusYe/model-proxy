## S1. Concrete Candidate Error Verification

### [Assertion 1] Task 1: `RoutingSnapshot.init(from:for:)` receives `AppConfig` which is a reference type (`@Observable final class`), but `RoutingSnapshot` is a `Sendable` struct. In Swift 6 strict concurrency, passing a non-Sendable `AppConfig` instance into a function that produces a `Sendable` value will cause a compile error if `AppConfig` is accessed across concurrency domains.

**Verification:** Read `ModelProxy/Models/AppConfig.swift:7-8` -- `@Observable final class AppConfig: Codable` does NOT conform to `Sendable`. The plan's Task 6 `StatusPopover.onAppear` creates the snapshot on `@MainActor` (the `Task {}` in onAppear inherits `@MainActor`), and `RoutingSnapshot.init` only reads from `AppConfig` synchronously within the same actor context. The snapshot is a value type and does not retain the `AppConfig` reference. However, `ProxyServer.start()` is `@MainActor` and receives the snapshot as a value -- no cross-isolation boundary for `AppConfig` itself.

**Result:** The assertion does not hold for the happy path. The `AppConfig` reference is only accessed on `@MainActor`, and the `RoutingSnapshot` value type is what crosses isolation boundaries. However, this reveals a **secondary concern**: `ClientConfig.modelMappings` maps `String -> UUID`, but the plan's `RoutingSnapshot.init` iterates `clientConfig.modelMappings` expecting `[String: UUID]` (model ID to vendor UUID). The plan code at line 71 does `for (modelID, vendorID) in clientConfig.modelMappings` and then looks up `config.vendors.first(where: { $0.id == vendorID })`. This is correct per `ClientConfig.swift:14` which declares `var modelMappings: [String: UUID]`. No issue.

**Result:** PASS -- no compilation issue.

### [Assertion 2] Task 4: `ProxyServer` is declared `@MainActor @Observable final class` but NOT as an `actor`. Task 6 injects it via `.environment(proxyServer)`. SwiftUI's `@Environment` requires the type to conform to `Observable` (it does) but the type also needs to be usable with `.environment()` which for custom types requires either the `@Observable` macro or explicit `EnvironmentKey`. The plan uses the shorthand `.environment(proxyServer)` which relies on the Swift 5.9+ `@Observable` environment injection.

**Verification:** `@Observable` classes can be injected via `.environment(instance)` and retrieved via `@Environment(ProxyServer.self)`. This is the correct SwiftUI pattern for `@Observable` classes. `ProxyServer` is `@MainActor` and `@Observable final class` -- this matches the pattern used for `AppConfig` already in the project (see `ModelProxyApp.swift:10` using `.environment(configStore.config)` and `StatusPopover` would use `@Environment(AppConfig.self)`).

**Result:** PASS. But note: the current `StatusPopover.swift` does NOT use `@Environment(AppConfig.self)` -- it has no environment dependency at all (it's a placeholder). The plan's new `StatusPopover` code adds `@Environment(AppConfig.self) private var config` which matches the existing `.environment(configStore.config)` injection in `ModelProxyApp.swift`. This is consistent.

### [Assertion 3] Task 3: `ProxyChannelHandler` is marked `@unchecked Sendable` and stores mutable state (`requestHead`, `bodyBuffer`). In Swift 6 strict concurrency, this is only safe if the handler is guaranteed to be accessed from a single NIO event loop thread. NIO guarantees this for `ChannelHandler`s. However, the `Task {}` launched from `channelRead` captures `self.router` and `self.httpClient` -- NOT `self`. The handler resets `requestHead` and `bodyBuffer` immediately after launching the Task. If a second request arrives on the same channel before the Task completes, the handler accumulates the new request while the old one is being forwarded. This is correct keep-alive behavior.

**Result:** PASS -- the design correctly separates accumulation state (in handler, on event loop) from forwarding (in Task, on cooperative pool).

### [Assertion 4] Task 6: The plan modifies `ConfigStore.loadOrCreateDefault()` to add a baseURL migration that strips trailing `/v1`. The migration code `config.vendors[i].baseURL.hasSuffix("/v1")` only checks for `/v1` suffix. But `Vendor.swift:8` documents baseURL as "without version path". The existing `AppConfig.makeDefault()` at line 48 has `baseURL: "https://api.anthropic.com/v1"` -- the plan correctly identifies this needs to change to `"https://api.anthropic.com"`. However, the migration only strips `/v1` -- what about vendors with baseURLs ending in `/v1/` (trailing slash) or `/v2`?

**Verification:** Read `AppConfig.swift:48` -- the default value is `"https://api.anthropic.com/v1"` (no trailing slash). The plan's migration uses `.hasSuffix("/v1")` which handles this exact case. User-entered URLs with trailing slash (`/v1/`) would NOT be caught, but the `ProxyForwarder` at plan line 313 does `baseURL.trimmingCharacters(in: CharacterSet(charactersIn: "/"))` before concatenation, which handles trailing slashes. The migration is a best-effort cleanup for the known default.

**Result:** PASS -- acceptable for Phase 2. Edge case of `/v1/` is handled by the trimming in ProxyForwarder.

### [Assertion 5] Task 5/Task 3: `ResponseRelay.relay()` and `ProxyForwarder.sendError()` both call `channel.close()` after writing the response. But `ProxyChannelHandler.channelRead` resets `requestHead`/`bodyBuffer` for keep-alive. If the channel is always closed, the keep-alive reset is dead code. More critically: `sendError()` at plan line 394 calls `channel.close()`, and then `ResponseRelay.relay()` at plan line 586 also always calls `channel.close()`. These are consistent -- Phase 2 always closes. However, there is a potential double-close scenario: `sendError` is called from `ProxyForwarder.forward` when routing fails, and the channel is closed. The `errorCaught` handler in `ProxyChannelHandler` (line 271) also calls `context.close()`. If `sendError`'s write fails and triggers `errorCaught`, the channel would be closed twice. NIO handles double-close gracefully (second close returns an already-closed error), so this is benign.

**Result:** PASS -- no functional issue. The keep-alive reset at line 265-266 is dead code in Phase 2 but causes no harm.

## S2. Failure Reverse Reasoning

### [Compile Failure 1] `ProxyServer` uses `@Observable` macro but is also `@MainActor`. In Swift 6, `@Observable @MainActor final class` should compile. But `ProxyServer` stores `RequestRouter` (an `actor`) as a property. Since `RequestRouter` is an `actor`, storing it in a `@MainActor` class is fine -- actors are `Sendable`. `HTTPClient` from AsyncHTTPClient is also `Sendable`. `MultiThreadedEventLoopGroup` is `Sendable`. `Channel` (as `any Channel`) -- NIO's `Channel` protocol inherits `Sendable`. All properties are either value types, `Sendable` actors, or optionals thereof.

**Plan coverage:** PASS -- no compile issue expected.

### [Compile Failure 2] `ProxyForwarder.forward()` calls `channel.writeAndFlush(NIOAny(...))` from an async context. In NIO 2.x, `Channel.writeAndFlush` returns `EventLoopFuture<Void>`. The plan uses `.get()` to bridge to async (e.g., plan line 385-386: `try? await channel.writeAndFlush(...).get()`). This is the correct NIO-to-async bridge pattern. However, `NIOAny` init requires the correct type -- for an `HTTPServerResponsePart`, the channel pipeline must have the HTTP codec installed. The plan's `ServerBootstrap` at line 465 calls `channel.pipeline.configureHTTPServerPipeline(withErrorHandling: true)` which adds the HTTP codec. The handler writes `HTTPServerResponsePart` which is the correct outbound type for a server-side HTTP pipeline.

BUT: the `ProxyForwarder.sendError` and `ResponseRelay.relay` write `NIOAny(HTTPServerResponsePart.head(...))` etc. through `channel.writeAndFlush`. This goes through the channel pipeline. The issue: `ProxyForwarder` writes through the `Channel` directly (not through `ChannelHandlerContext`). Writing through `Channel` means the write traverses the entire pipeline from tail to head. The `ProxyChannelHandler` is typed with `OutboundOut = HTTPServerResponsePart`, and the HTTP encoder is earlier in the pipeline. Writing `HTTPServerResponsePart` through the channel should work because the HTTP response encoder in the pipeline handles this type.

**Plan coverage:** PASS -- the pipeline is correctly configured.

### [Compile Failure 3] The plan's `StatusPopover` at line 703 accesses `config.clients.first?.port`. `AppConfig` is `@Observable` but its `clients` property is `[ClientConfig]`. Accessing `config.clients.first` in an `@Observable` context should trigger observation tracking. No compile issue expected.

**Plan coverage:** PASS.

### [Runtime Regression 1] App Sandbox is enabled (`ModelProxy.entitlements` has `com.apple.security.app-sandbox = true`). The entitlements already include `com.apple.security.network.server` and `com.apple.security.network.client`, which are required for binding to localhost and making outbound HTTP requests. Without these, the server would fail to bind silently or with a cryptic error.

**Verification:** Read `ModelProxy.entitlements:6-10` -- both `network.server` and `network.client` are `true`.

**Plan coverage:** PASS -- entitlements are already correct. The plan does not need to modify them.

### [Runtime Regression 2] The plan starts the proxy in `StatusPopover.onAppear` -- meaning the proxy does NOT start until the user clicks the menu bar icon. If a client (Claude Code) sends a request before the user clicks the icon, the connection is refused. This is documented as a "Phase 2 limitation" at plan line 665, so it's intentional.

**Plan coverage:** PASS -- documented limitation.

### [Runtime Regression 3] `ProxyForwarder.forward()` sets both `Authorization: Bearer {key}` AND `x-api-key: {key}` headers (plan lines 332-333) on ALL upstream requests, regardless of vendor. Non-Anthropic vendors (e.g., DashScope/Alibaba) typically use only `Authorization: Bearer` and may reject requests with unknown `x-api-key` headers. For Phase 2 where the primary use case is Anthropic passthrough, this works. But when routing to non-Anthropic vendors, the extra `x-api-key` header could cause issues.

**Plan coverage:** NOT COVERED. The plan does not mention this limitation. For Phase 2 (primarily Anthropic passthrough), this is unlikely to cause failures, but it's a known gap for vendor routing.

**Severity:** Recommend -- not blocking for Phase 2 since the plan's verification (Task 8) only tests Anthropic endpoints.

### [Runtime Regression 4] The `baseURL` migration in ConfigStore only runs at load time. If a user launches the app, the migration runs. But the `AppConfig.makeDefault()` change (removing `/v1`) happens in the source code. If an existing user already has a `config.json` with the old default, the migration handles it. If a new user launches for the first time, `makeDefault()` produces the new correct URL. The sequence is correct.

**Plan coverage:** PASS.

### [Runtime Regression 5] `ContentView.swift` and `Item.swift` are Xcode template files that import SwiftData. The CLAUDE.md constraint says "No SwiftData, no Core Data". These files are not referenced by the plan and are not used by the current app entry point (`ModelProxyApp` uses `MenuBarExtra`, not `WindowGroup` with `ContentView`). However, their presence means SwiftData is linked into the binary. The plan does not address cleaning up these template files.

**Plan coverage:** NOT COVERED. These are pre-existing files outside the plan's scope. Not a regression introduced by this plan.

**Severity:** Noted but not blocking.

## Supplementary Findings

### [Finding 1] `AppConfig` is NOT `Sendable`

`AppConfig` at `ModelProxy/Models/AppConfig.swift:8` is `@Observable final class AppConfig: Codable` -- it does not conform to `Sendable`. In the plan, `RoutingSnapshot.init(from:for:)` takes `AppConfig` as a parameter. Since this init is called from `@MainActor` context (in StatusPopover.onAppear's Task which inherits MainActor), and `RoutingSnapshot` only reads from `AppConfig` synchronously within the init, this works. But if `RoutingSnapshot` were ever constructed off the main actor, Swift 6 strict concurrency would flag `AppConfig` as non-Sendable crossing an isolation boundary.

**Severity:** Not blocking for Phase 2. The plan correctly constrains construction to `@MainActor`.

### [Finding 2] `Proxy/` directory does not exist yet

The plan creates files in `ModelProxy/Proxy/` but this directory does not exist. The plan does not explicitly say "create the directory". However, any file-creation tool or manual file creation will create intermediate directories. With Xcode's `PBXFileSystemSynchronizedRootGroup`, new files in the `ModelProxy/` tree are auto-discovered.

**Severity:** Not blocking -- directory creation is implicit in file creation.

### [Finding 3] No `Proxy/` directory in Xcode project structure

The project uses `PBXFileSystemSynchronizedRootGroup` for the `ModelProxy/` directory (confirmed in `project.pbxproj`). New files placed in `ModelProxy/Proxy/` will be automatically picked up by Xcode without manual project file editing.

**Severity:** Not blocking -- confirmed auto-discovery.

### [Finding 4] App Sandbox and file access for config migration

The migration in Task 6 modifies config loaded from `~/Library/Application Support/ModelProxy/config.json`. With App Sandbox enabled, the app's container is redirected. `FileManager.urls(for: .applicationSupportDirectory)` returns the sandboxed path. The existing `ConfigStore` already reads/writes from this location successfully (Phase 1 is working). The migration adds in-memory transformation before returning -- no additional file access needed.

**Severity:** Not blocking.
