### [Compile Failure 3] Task 5 migration -- `ClientConfig.defaultUpstream.isEmpty` check assumes empty string fallback works, but `Codable` ignores unknown keys only by default for structs

**Assumed failure:** Task 5's migration says "Old `Vendor` objects may have a `modelPatterns` key in JSON: harmless since `Vendor` no longer has the field and `Codable` ignores unknown keys by default." This is correct for Swift's default `Codable` synthesis. However, for `ClientConfig`, Task 5 adds a custom `init(from:)` that uses `try?` for `defaultUpstream` with `?? ""` fallback. Old JSON has `modelMappings` key but no `defaultUpstream` key. The custom decoder correctly handles this.

But there's a subtlety: old `ClientConfig` JSON also has a `modelMappings` key. The new custom decoder does NOT decode `modelMappings` (it's not in the new CodingKeys). Swift's `KeyedDecodingContainer` ignores unknown keys, so `modelMappings` in the JSON is silently skipped.

**Plan coverage:** COVERED. The `try?` with `?? ""` fallback for `defaultUpstream` handles the missing key case. Unknown keys (`modelMappings`) are ignored by `KeyedDecodingContainer`.

### [Runtime Regression 1] Task 12: Port change fires `saveAndReload` on every keystroke

**Assumed regression:** The Clients tab's port field uses `.onChange(of: portText)` which fires on every character typed. Typing "8082" fires 4 times: "8", "80", "808", "8082". The guard `Int(newValue), (1024...65535).contains(port)` rejects "8" (< 1024), "80" (< 1024), "808" (< 1024), but accepts "8082". Only one save fires. This is correct but the UX is poor if the user types a port like "9000" -- "9" is rejected (< 1024), "90" rejected, "900" rejected, "9000" accepted. Only one save fires. This is actually fine.

However, if the user is editing an existing port (e.g., changing 8080 to 9090), they might clear the field first. Typing backspace 4 times produces "", then typing "9090". The empty string is rejected by `Int(newValue)`. Intermediate states "9", "90", "909" are all rejected. Only "9090" triggers save. This is correct.

**Plan coverage:** COVERED. The guard clause prevents invalid intermediate saves.

## Suggested Improvements (risks exposed during verification)

### 1. [Task 12] DP-002 banner not implemented
**Source:** S2 Runtime Regression 2

The plan's DP-002 decision says "show a banner in the Clients tab ('Port change takes effect after stopping and restarting the proxy') when port differs from currently bound port." Task 12's code does not include this banner. Users will have no indication that their port change requires a proxy restart.

### 2. [Task 8] Partial start error handling
**Source:** S2

If one client port fails to bind (e.g., port in use), the proxy starts with remaining clients and surfaces the error via `lastError`. The StatusPopover shows this as an error state. Consider whether the UI should distinguish between "fully running" and "partially running" states, since `isRunning == true` with `lastError != nil` is a partial start.

### 3. [Task 12] `saveAndReload` called on every keystroke for defaultUpstream field
**Source:** S2 analysis

The defaultUpstream TextField fires `set:` on every character typed. Unlike the port field which has a `guard` rejecting most intermediate values, the defaultUpstream field saves on every keystroke (e.g., typing "https://api.anthropic.com" triggers 30+ saves and routing snapshot swaps). Consider using `.onSubmit` or debouncing.
